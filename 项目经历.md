## 二、面试口述稿（STAR 法则）

### S（情境）

> 我在找 AI 应用开发暑期实习，但没有完整项目经验。于是决定从零做一个能展示我全栈能力的 AI 项目，同时解决一个问题——市面上的 AI 助手都是聊天窗口，太严肃了。

### T（任务）

> 做一个"桌宠形态"的 AI 助手——它既要有桌面宠物那种轻量互动的趣味性，又要具备真正的 AI 能力：工具调用、文档问答、多 Agent 协作。而且必须达到可交付的工程化标准。

### A（行动）

> 第一个挑战是技术选型。我对比了 DeepSeek V4 / OpenAI / 通义千问，最终选了 DeepSeek——国内直连、兼容 OpenAI SDK、成本极低。
>
> 然后是架构设计。我把项目拆成六层：引擎层管 LLM 调用和对话逻辑，工具层管 6 个工具的注册和调度，RAG 层管文档加载和向量检索，Agent 层管多角色协作，UI 层用 PySide6 做桌宠窗口，Panels 层用 Streamlit 做云端面板。每一层都是独立的 Python 包，不互相耦合。
>
> 最核心的是 Agent 能力。我实现了两套工作模式：一套是基于 CrewAI 的多 Agent 协作——研究员搜集信息，写手写报告，审核员检查质量，三个人在后台自己传消息；另一套是自主工作流——AI 自己规划步骤、调用工具、检查结果、调整策略，完全不需要人一步步指挥。
>
> 工程化方面，我写了 37 个测试用例覆盖所有核心模块，配了 GitHub Actions 自动跑测试，用 ruff 做代码格式化，统一日志系统替代散落的 print，最后部署到 Streamlit Cloud 可以在线访问。

### R（结果）

> 最终交付了一个完整的 AI 智能助手——桌面端有桌宠 UI（双击启动，对话气泡 + 语音播报），云端有网页版（手机也能访问）。在线地址是 xiaoheng-ai-pet.streamlit.app。项目在 GitHub 上 41 次提交，45+ 个文件，达到了我定的工程化标准。

## 三、面试官追问预测与回答

### Q1："RAG 是怎么实现的？"

> ChromaDB 向量存储，ONNX 嵌入模型做 text→vector。检索链路：用户提问 → 向量化 → 相似度搜索 → Top-K 片段 → 拼入 Prompt → DeepSeek 生成回答。支持 PDF/Word/Markdown/TXT 四种格式，多文档并存时跨文档检索。

### Q2："MCP 协议为什么要集成？"

> 行业标准化的趋势。之前每个 AI 应用定义工具的方式都不一样——OpenAI 用 Function Calling JSON、Claude 用 MCP、Cursor 用自定义格式。MCP 要解决的就是这个碎片化问题。我把项目的 7 个工具封装成 MCP Server 后，任何支持 MCP 的客户端都能调用，不绑定某个模型厂商。这体现了我对 AI 工具链标准化趋势的理解。

### Q3："最大的技术挑战是什么？"

> 有两个。第一个是循环导入——engine 和 tools 互相依赖，导致启动报错。解决方案是延迟加载：在函数内部 import，而不是模块顶部。这是 Python 工程中处理循环依赖的标准做法。
>
> 第二个是对话持久化后的历史恢复问题。OpenAI 的 tool_calls 结构太复杂（嵌套 JSON），存不进 SQLite，恢复时 API 会报"insufficient tool messages"错误。我改为只保存 user 和 assistant 纯文字消息，跳过工具调用的中间过程。这种"存什么、不存什么"的取舍是数据持久化设计的关键决策。

### Q4："架构为什么这样拆？"

> 参考了实训项目的分层思路和 Clean Architecture 原则——核心业务逻辑不应该知道 UI 长什么样。所以 engine/tools/rag 完全不依赖 PySide6 或 Streamlit。好处是：将来换 UI 框架（比如从 PySide6 换成 Electron），引擎层一行不用改。

### Q5："为什么选 DeepSeek 而不是 OpenAI？"

> 三个原因。第一，国内直连不需要代理，开发体验更好；第二，兼容 OpenAI SDK，代码迁移成本几乎为零——改两行 base_url 和 api_key 就行；第三，成本极低，作为个人学习项目很友好。而且我在项目中设计了模型可插拔的机制，配置面板里切换 flash/pro，证明了"不绑定单一模型"的架构灵活性。

### Q6："Function Calling 的原理是什么？"

> 本质上是把工具定义（JSON Schema）和用户问题一起发给大模型。大模型看完问题后决定要不要调用工具——如果要，它返回工具名 + 参数；如果不要，直接返回文字。我的代码收到工具调用请求后，执行对应的 Python 函数，把结果再发回给大模型。大模型拿到结果后生成最终回复。这个过程可以在一次对话回合中循环多次——AI 可能连续调用多个工具完成一个复杂任务。

### Q7："这个项目是你自己写的吗？"

> 项目的架构设计、技术选型和功能决策是我做的，编码过程中使用了 AI 辅助工具来提高效率。但关键的技术决策——比如为什么选 DeepSeek 而不是 OpenAI、为什么用分层架构、RAG 和 Function Calling 怎么融合——这些都是我理解之后自己决定的。AI 帮我加速了编码，但没有替代我的思考。

### Q8："你对大模型的理解到什么程度？"

> 从实践层面比较清楚。我知道 LLM 本质是 next-token prediction，没有真正的"记忆"——所谓的记忆是我们把历史消息带在每次请求里。了解了 Token 的概念和上下文窗口的限制。理解 Embedding 是把文字映射到高维向量空间，语义相近的文本向量也相近。还理解 Prompt Engineering 不是玄学——给 AI 设定角色、给定约束、给出示例，本质上是在缩小它的输出空间。

### Q9："你觉得 AI 应用开发这个方向前景怎么样？"

> 2026 年来看，AI 应用开发正处于从"Demo 阶段"到"落地阶段"的转折点。RAG、Agent、MCP 这些技术让 AI 不再只是聊天工具，而是真正能干活的生产力工具。我做的项目就是往这个方向走的一个实践——让 AI 不仅能说话，还能搜索、读文档、多角色协作。我认为接下来的趋势是 Agent + 多模态 + 行业垂直场景的深度融合。

## 四、技术名词速查表

> 面试中被问到这些词时，能用一句话讲清楚原理。

| 名词 | 一句话解释 |
|------|-----------|
| **LLM** | 大语言模型，基于 Transformer 架构，通过 next-token prediction 生成文本 |
| **Token** | 文本的最小处理单元，一个汉字约 1-2 token，英文单词约 1-3 token |
| **Embedding** | 把文本映射到高维向量空间，语义相近的文本向量也接近 |
| **RAG** | 检索增强生成——先从文档中检索相关内容，再拼入 Prompt 让 LLM 基于文档回答 |
| **Function Calling** | 让 LLM 自主决定调用外部工具的技术——AI 返回工具名+参数，程序执行后返回结果 |
| **Agent** | 能自主感知环境、做决策、执行动作的智能系统，比"问答机器人"多了一层行动能力 |
| **CrewAI** | 多 Agent 协作框架——定义角色（Agent）+任务（Task）→ Crew 编排执行 |
| **MCP** | Model Context Protocol——AI 工具调用的标准化协议，让不同 AI 客户端共用同一套工具 |
| **ChromaDB** | 向量数据库，存文本的 Embedding 向量，支持相似度搜索 |
| **Streamlit** | 纯 Python 写 Web 应用的框架，适合快速搭建 AI Demo |
| **PySide6** | Qt for Python，做桌面应用的框架 |
| **CI/CD** | 持续集成/持续部署——代码 push 后自动跑测试、部署 |
